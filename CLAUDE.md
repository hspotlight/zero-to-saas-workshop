# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Repository Is

A **Claude Code skills workshop** — not a runnable application. It contains skill definitions that scaffold and guide SaaS project development. The skills are invoked via Claude Code's `/` commands.

## Available Skills

| Skill | Trigger | Purpose |
|---|---|---|
| `firebase-webapp-scaffold` | `/firebase-webapp-scaffold` | Interview user and scaffold a complete Firebase web app |
| `frontend-design` | `/frontend-design` | Generate distinctive, production-grade UI (plugin) |
| `tdd` | `/tdd` | Red-green-refactor TDD loop for features/bug fixes |
| `grill-me` | `/grill-me` | Stress-test a plan through relentless questioning |
| `to-prd` | `/to-prd` | Convert conversation context into a PRD GitHub issue |
| `qa` | `/qa` | Conversational QA session that files GitHub issues for bugs |

Skills/plugins live in:
- `.claude/skills/` — local skills defined in this repo
- `.agents/skills/` — imported external skills (from `mattpocock/skills`)
- Plugins — installed via `/plugin install` (e.g. `frontend-design@claude-code-plugins`)

`skills-lock.json` tracks skill versions (like a lockfile for skill imports).

## firebase-webapp-scaffold: What It Produces

This is the primary skill. When invoked, it **interviews the user first** (never jumps straight to code), then scaffolds a full project. The scaffolded project's stack:

- **Frontend**: Plain HTML/CSS/JS — no framework, Firebase SDK loaded from CDN (compat mode v10.8.0); optionally styled with `frontend-design`
- **Auth**: Firebase Authentication (email/password)
- **Database**: Cloud Firestore — all data under `users/{userId}/` by default
- **Hosting**: Firebase Hosting
- **Testing**: Jest + jsdom
- **CI/CD**: GitHub Actions with OIDC (Workload Identity Federation — no stored secrets)
- **Package manager**: pnpm

### Commands in scaffolded projects

```bash
pnpm install          # Install deps
pnpm serve            # Copy test config + serve on localhost:3000
pnpm test             # Run Jest tests
pnpm test:watch       # Watch mode
pnpm deploy           # Firebase deploy
```

### Architecture of scaffolded projects

```
{app-name}/
├── public/
│   ├── utils.js              # Firebase init + all Firestore CRUD helpers
│   ├── login.js              # Auth logic (signIn, register, onAuthStateChanged)
│   ├── {feature}.js          # One JS file per major feature/page
│   ├── style.css
│   ├── firebase-config.js    # Gitignored — generated by copying from config/
│   └── config/
│       ├── firebase-test.js  # Test project config (committed, not secret)
│       └── firebase-prod.js  # Prod project config (committed, not secret)
├── __tests__/
│   ├── setup.js              # Global Firebase mock for Jest
│   └── {feature}.test.js
└── .github/workflows/
    ├── deploy-test.yml       # Deploys to test on PR
    └── deploy-prod.yml       # Deploys to prod on push to main
```

Key principles enforced by the scaffold:
- **User-isolation**: Every Firestore path starts with `users/{userId}/`; security rules enforce this
- **Config strategy**: Environment configs are committed (not secrets); active `firebase-config.js` is gitignored and copied at build/serve time
- **XSS prevention**: `escapeHtml()` used before inserting any user content into the DOM
- **Tests target pure functions**: No Firebase SDK calls in tests — validation, data shapes, and UI helpers only

## tdd Skill Philosophy

- Tests verify behavior through **public interfaces**, not implementation details
- Use **vertical slices** (tracer bullet through the stack), not horizontal layers
- Prefer integration-style tests over unit tests
- Avoid mocking internals — mock only at system boundaries
